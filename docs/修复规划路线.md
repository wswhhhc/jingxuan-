# 修复规划路线（需求/规划/代码对齐）

## 目标

围绕当前已识别的三类问题推进修复：

1. 规划已写但代码未闭环
2. 需求已写但规划拆解不完整
3. 已实现但前后端字段/链路未打通

原则：

- 先修 P0 主链路，再补管理能力和体验问题
- 每一阶段必须带验收标准，避免“文档完成、代码未闭环”
- 每一阶段都配一条可直接执行的 Prompt，方便分配给 AI / 组员

---

## 阶段 1：先修作品附件链路

### 目标

让“上传附件 -> 保存作品 -> 编辑作品 -> 查看详情”形成真实闭环，修复当前“文件已上传但未关联到作品”的问题。

### 涉及问题

- 上传接口未返回附件实体 ID
- 前端提交作品时未传 `attachmentIds`
- 编辑作品时附件重绑逻辑存在，但前端未喂给后端所需字段

### 交付物

- 上传接口返回可用于作品关联的附件 ID
- 前端保存草稿/提交审核/编辑作品时正确传递 `attachmentIds`
- 作品详情、我的作品列表可看到已关联附件

### 验收标准

- 新建作品后上传的附件可在详情页展示
- 编辑作品后增删附件能正确反映到数据库
- 未关联的临时上传文件不会被误认为作品正式附件

### Prompt

```text
你现在负责修复“作品附件上传与关联链路”。请只处理附件相关问题，不改评分、排行、评论模块。

目标：
1. 后端上传接口返回附件记录 ID，而不只是 URL
2. 前端创建作品、更新作品时，把附件列表转换为 attachmentIds 传给后端
3. 后端创建/更新作品时，正确把这些 attachmentIds 关联到 work_attachment.work_id
4. 学生作品详情页和我的作品页能看到已关联附件

必须先检查这些文件：
- backend/src/main/java/com/jingxuan/modules/work/controller/FileUploadController.java
- backend/src/main/java/com/jingxuan/modules/work/service/impl/WorkServiceImpl.java
- backend/src/main/java/com/jingxuan/modules/work/dto/WorkCreateRequest.java
- backend/src/main/java/com/jingxuan/modules/work/dto/WorkUpdateRequest.java
- frontend/src/api/student/work.ts
- frontend/src/views/student/WorkSubmit.vue

输出要求：
- 直接修改代码
- 保持现有接口风格
- 最后给出“修改点 + 风险点 + 如何验证”
```

## 阶段 2：修复评分批次主链路

### 目标

让作品在创建/提交时真正绑定评分批次，使“评分有效期、班级范围、排行榜、公示、学生查看我的排名”都依赖真实数据运行。

### 涉及问题

- 创建作品时查了活跃批次，但未写入 `work.batch_id`
- 排行榜、公示通知、学生我的排名都依赖 `batch_id`
- 当前导致批次功能看似存在，实际可能退化或静默失效

### 交付物

- 作品创建时正确写入活跃批次 ID
- 必要时补充编辑/提交场景对批次的处理规则
- 排行榜、公示、学生“我的排名”链路基于真实 `batch_id` 运行

### 验收标准

- 新建作品后数据库 `work.batch_id` 不为空
- 教师评分时能正确命中批次时间和班级范围校验
- 公示排行榜后，学生端“我的排名”有数据
- 历史批次排行榜筛选可返回对应批次数据

### Prompt

```text
你现在负责修复“评分批次主链路”。重点是让作品和批次真实绑定，并让后续评分/排行/公示链路恢复正常。

目标：
1. 修复 Work 创建流程，确保作品落库时写入正确的 batchId
2. 检查 submitWork、updateWork、score submit、public ranking、student my-ranks 是否都依赖真实 batchId
3. 如果存在“作品跨批次”的业务规则缺口，按当前需求文档做最小闭环，不做额外大改
4. 修复后保证评分批次、公示排行榜、学生查看我的分数排名可以跑通

必须先检查这些文件：
- backend/src/main/java/com/jingxuan/modules/work/service/impl/WorkServiceImpl.java
- backend/src/main/java/com/jingxuan/modules/score/service/impl/ScoreServiceImpl.java
- backend/src/main/java/com/jingxuan/modules/scorebatch/service/impl/ScoreBatchServiceImpl.java
- backend/src/main/java/com/jingxuan/modules/rank/controller/PublicRankController.java
- backend/src/main/java/com/jingxuan/modules/adapter/StudentApiController.java
- backend/src/main/resources/mapper/business/WorkMapper.xml

约束：
- 不要顺手重构 unrelated 代码
- 优先修真实业务闭环
- 修改后说明“原来为什么失效、现在为什么能通”

最后输出：
- 修改文件
- 业务影响
- 验证步骤
```

## 阶段 3：修复评论展示与回复链路

### 目标

让评论模块从“接口能返回数据”提升到“页面字段正确、评论人正确显示、回复关系正确展示”。

### 涉及问题

- 后端返回 `userName`，前端读取 `username`
- 前端期待 `replies` 树形结构，后端目前主要返回平铺分页列表
- 评论模块已部分实现，但展示层未真正闭环

### 交付物

- 评论列表字段统一
- 回复数据结构与前端展示方式统一
- 评论区正确显示评论人姓名、回复关系、时间

### 验收标准

- 评论列表不再显示“匿名用户”占位
- 主评论与回复能按预期展示
- 发表评论、回复评论后刷新可正确看到结果

### Prompt

```text
你现在负责修复“评论展示与回复链路”。请只关注评论模块，不改附件、批次、排行。

目标：
1. 统一评论 DTO 和前端类型字段，解决 userName / username 不一致
2. 明确评论列表返回结构：如果前端展示需要树形 replies，就在后端组装；如果保持平铺，就同步调整前端
3. 修复作品详情页评论区，让评论人、回复关系、评论时间都正确展示

必须先检查这些文件：
- backend/src/main/java/com/jingxuan/modules/comment/dto/CommentVO.java
- backend/src/main/java/com/jingxuan/modules/comment/service/impl/CommentServiceImpl.java
- backend/src/main/java/com/jingxuan/modules/comment/controller/CommentController.java
- frontend/src/api/public/comment.ts
- frontend/src/views/public/WorkDetail.vue

要求：
- 选一种最稳的评论返回结构，不要两边都半兼容
- 保持已有评论发布接口可用
- 最后说明“字段契约如何统一、回复结构如何组织”
```

## 阶段 4：补齐后台“操作日志 / 数据字典”规划与页面闭环

### 目标

把目前只做到后端接口的“操作日志”和“数据字典”补成真正可用的后台能力。

### 涉及问题

- 需求里这两个模块是独立功能
- 规划只写了“接口”，没拆页面交付
- 当前后台路由中没有对应入口

### 交付物

- `操作日志` 管理页
- `数据字典` 管理页
- 后台菜单/路由补齐

### 验收标准

- 管理员可查看关键操作日志
- 管理员可维护技术栈、班级、作品类型等字典项
- 后台有可点击入口，不是“只有接口没有页面”

### Prompt

```text
你现在负责补齐“操作日志 + 数据字典”后台闭环。当前后端已有基础接口，但前端页面和路由不完整。

目标：
1. 为操作日志补一个管理员页面，支持分页查看、按 action 简单筛选
2. 为数据字典补一个管理员页面，支持按类型查看、新增、编辑、删除
3. 补齐后台路由和入口，让管理员可直接访问

必须先检查这些文件：
- backend/src/main/java/com/jingxuan/modules/log/controller/LogController.java
- backend/src/main/java/com/jingxuan/modules/dict/controller/DictController.java
- frontend/src/router/modules/admin.ts
- frontend/src/layout/AdminLayout.vue

实现要求：
- 复用现有 Element Plus 页面风格
- 先做最小可用，不追求复杂高级筛选
- 不修改无关业务模块

最后输出：
- 新增页面
- 路由入口
- 验证方式
```

## 阶段 5：补一轮非功能需求落地与验收

### 目标

把现在停留在“文档写了、只部分冒烟”的非功能要求，补成可交付的验收清单与必要实现。

### 涉及问题

- 目前主要补了构建与基础冒烟
- 需求里还有浏览器兼容、响应时间、并发目标、上传安全、配置安全等要求
- 还需要确认 DeepSeek 配置、连通性测试、模型名与需求是否一致

### 交付物

- 一份非功能验收清单
- 必要的配置对齐修复
- 若无法一次实现压测，至少补“验证方式 + 风险说明”

### 验收标准

- 上传安全规则可说明并验证
- `DEEPSEEK_API_KEY` 走环境变量
- 连通性测试按钮真实可用
- 模型配置与需求文档达成一致，或在文档中记录偏差原因

### Prompt

```text
你现在负责“非功能需求落地与验收补齐”。不要大改业务功能，重点做配置、验收、风险收口。

目标：
1. 对照需求文档中的非功能要求，列出哪些已满足、哪些未满足、哪些只能记录风险
2. 检查 DeepSeek 配置是否符合需求：环境变量、默认模型、连通性测试、前端不暴露 Key
3. 补一份可执行的非功能验收清单，优先覆盖上传安全、权限、公开访问、内容审核配置

必须先检查这些文件：
- docs/需求文档.md
- docs/task_plan.md
- backend/src/main/resources/application.yml
- backend/src/main/java/com/jingxuan/config/DeepSeekConfig.java
- backend/src/main/java/com/jingxuan/modules/sensitive/service/impl/DeepSeekReviewServiceImpl.java
- backend/src/main/java/com/jingxuan/modules/sensitive/controller/SensitiveRuleController.java
- frontend/src/views/admin/rule/index.vue
- frontend/src/api/admin/rule.ts

要求：
- 区分“代码修复”“文档补录”“暂不处理但要记风险”
- 不凭感觉说通过，要给出验证依据
- 最后输出一个清晰的验收状态表
```

## 阶段 6：回归测试与文档回写

### 目标

在前 5 个阶段完成后，统一做一轮链路回归，把结果回写到规划和进度文档，避免再次出现“文档已完成但实际未通”。

### 覆盖链路

- 学生：登录 -> 创建作品 -> 上传附件 -> 提交审核
- 管理员：审核 -> 发布 -> 配置批次 -> 公示排行榜
- 教师：评分 -> 修改评分 -> 查看排行榜
- 前台：作品列表 -> 详情 -> 评论 -> 排行榜
- 学生：查看我的排名

### 交付物

- 一份回归测试记录
- 更新后的 `docs/task_plan.md`
- 更新后的 `docs/progress.md`
- 如有新增问题，补到 `docs/findings.md`

### 验收标准

- 每条核心链路至少有一次真实验证结果
- 文档中的“已完成”状态与真实代码一致
- 遗留问题有明确阻塞说明，不再用“默认已完成”带过

### Prompt

```text
你现在负责“修复后的统一回归测试与文档回写”。不要继续开发新功能，重点是验证和同步文档状态。

目标：
1. 逐条验证核心业务链路：附件、审核、评分批次、排行榜、公示、评论、学生我的排名
2. 把真实验证结果更新到 docs/task_plan.md、docs/progress.md、docs/findings.md
3. 对每个模块明确标记：已验证通过 / 已实现未验证 / 仍存在缺口

要求：
- 不要把“静态阅读代码”写成“已通过”
- 如果某条链路没法完整验证，要写清原因和阻塞点
- 最终给出一份按模块分组的回归结论
```

## 推荐执行顺序

1. 阶段 1：附件链路
2. 阶段 2：评分批次主链路
3. 阶段 3：评论展示链路
4. 阶段 4：日志 / 字典后台闭环
5. 阶段 5：非功能验收补齐
6. 阶段 6：统一回归与文档回写

原因：

- 阶段 1、2 是 P0 主链路，优先级最高
- 阶段 3、4 属于“已部分实现但未闭环”
- 阶段 5、6 负责把系统从“能写代码”推进到“能交付、能答辩、能复盘”

---

## 分工建议

### AI / 开发并行建议

- AI-1：阶段 1 附件链路
- AI-2：阶段 2 批次与排行榜主链路
- AI-3：阶段 3 评论链路
- AI-4：阶段 4 日志 / 字典后台页
- 组长：阶段 5 非功能验收 + 阶段 6 回归文档回写

### 合并顺序建议

1. 先合并阶段 1、2
2. 再合并阶段 3、4
3. 最后统一做阶段 5、6

这样可以避免回归时被未完成主链路干扰。
