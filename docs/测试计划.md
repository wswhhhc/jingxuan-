## 1. 测试范围与目标

### 1.1 测试范围

本项目为前后端分离的全栈应用，测试覆盖以下维度：

| 层次 | 范围 | 测试类型 |
|------|------|----------|
| 后端 Service | 18 个 Service 实现类的核心业务逻辑 | 单元测试 |
| 后端 Controller | 4 个 Adapter 控制器 + 其他控制器，约 80+ 个 API 端点 | 集成测试 |
| 前端组件 | 28 个路由对应 Vue 组件 | 组件测试 |
| 前端 API 层 | 所有 API 调用函数的枚举映射、VO 适配、分页适配 | 适配层测试 |
| 端到端流程 | 作品生命周期、评分、审核、点赞、评论等 7 条主线 | 集成/E2E 测试 |

### 1.2 测试目标

- 验证所有 API 的功能正确性（正向 + 异常）
- 验证核心业务流程的完整性和一致性
- 验证权限控制的有效性（RBAC 四角色隔离）
- 验证前端数据适配层的正确性（枚举映射、字段适配）
- 验证异常场景的容错能力（参数校验、边界值、外部依赖失败）

### 1.3 测试角色覆盖

测试必须覆盖以下 4 种角色：

| 角色 | 覆盖要求 |
|------|----------|
| **管理员（ADMIN）** | 审核管理、用户管理、角色菜单、字典、日志、公告、奖项、端口、敏感规则、评分批次 |
| **教师（TEACHER）** | 评分、排行榜查看、已审核作品列表、通知 |
| **学生（STUDENT）** | 作品 CRUD、提交审核、查看排名/成绩、通知 |
| **匿名访客（公开端）** | 浏览已发布作品列表/详情、查看排行榜、评论列表 |

### 1.4 测试优先级

| 优先级 | 范围 | 说明 |
|--------|------|------|
| `P0` | 登录认证、作品创建与提交、审核、评分、发布、公开展示 | 核心主链路，必须优先完成并通过 |
| `P1` | 评论、点赞、排行榜、文件上传、通知、筛选查询 | 重要功能，影响主要使用体验 |
| `P2` | 字典、公告、日志、角色菜单、端口、奖项、敏感规则等后台支撑功能 | 支撑性功能，可在核心链路稳定后补齐 |

## 2. 测试策略

### 2.1 单元测试

- **框架**：JUnit 5 + Mockito
- **范围**：Service 实现类，隔离外部依赖（数据库、Redis、DeepSeek API）
- **方法**：Mock Mapper 和外部调用，验证业务逻辑分支
- **目标覆盖**：每个 Service 的核心方法至少一个正向 + 一个异常用例

### 2.2 集成测试

- **框架**：Spring Boot Test + MockMvc（或 WebTestClient，如使用 WebFlux）
- **范围**：Controller 层，带真实 Service 但 Mock 外部依赖
- **方法**：启动 Spring 上下文（`@SpringBootTest`），通过 MockMvc 发送 HTTP 请求验证响应
- **数据库**：使用 H2 内存数据库或独立测试 MySQL 实例

### 2.3 前端测试

- **框架**：Vitest + `@vue/test-utils`
- **范围**：Vue 组件渲染和交互、路由守卫、API 适配层
- **方法**：挂载组件、模拟用户操作、断言 DOM 输出

### 2.4 端到端测试

- **范围**：跨前后端的完整业务流
- **方法**：API 级别模拟完整流程（后端集成测试覆盖），前端页面流（手动验证或 Playwright）

### 2.5 风险与假设

- DeepSeek 审核在测试环境中可通过 `fallback=bypass` 跳过
- Redis 在测试环境中允许保留连接，但业务代码需支持降级
- H2 若存在兼容性问题，可切换独立测试 MySQL 执行集成测试
- 时间不足时优先覆盖核心业务流（作品→审核→评分→发布）

### 2.6 准入标准

满足以下条件后方可进入正式测试执行：

- 需求范围、测试范围与角色权限边界已确认
- 测试计划文档已评审通过
- `application-test.yml`、测试数据库、Redis、必要环境变量已准备完成
- 测试数据库可按既定顺序完成基础表、业务表、种子数据和迁移 SQL 初始化
- 至少提供管理员、教师、学生、匿名访客对应的测试账号或等价认证方式
- 后端服务、前端页面、文件上传、登录认证等核心能力已可启动并完成基本联调
- 已明确 DeepSeek、Redis 等外部依赖在测试环境中的替代或降级策略

### 2.7 准出标准

满足以下条件后方可判定本轮测试通过并进入验收或发布决策：

- `P0` 缺陷全部关闭，`P1` 缺陷无阻断项且已评估风险
- 7 条端到端核心业务流程全部执行完成，且通过率为 100%
- 认证、权限、审核、评分、发布等核心模块无遗留阻断问题
- 已完成计划范围内的测试执行记录、缺陷报告和测试总结报告
- 遗留问题、未覆盖项、环境限制和已知风险已在测试总结中明确说明并获确认

## 5. 测试设计与用例

### 5.1 正向测试

每个 API 至少包含一个正向用例，验证：

- HTTP 状态码正确（200/201）
- 返回 `Result<T>` 的 `code` 为 200
- 返回数据结构和字段值符合预期

### 5.2 异常测试

每个 API 至少包含一个异常用例，覆盖：

- 参数校验失败（必填缺失、格式错误、超出边界）
- 权限不足（角色不匹配、未登录）
- 业务规则冲突（重复提交、越权操作）
- 资源不存在（404）

### 5.3 边界测试

- 分页参数边界值（page=0, page=-1, size=10000）
- 字符串长度边界
- 枚举值越界
- 并发操作

### 5.4 作品状态流转矩阵

| 状态 | 学生 | 教师 | 管理员 | 公开端 |
|------|------|------|--------|--------|
| 草稿 | 可查看、编辑、删除、提交审核 | 不可见 | 可按管理规则查看 | 不可见 |
| 已提交 | 可查看，不可重复提交 | 不可评分 | 可审核、可驳回、可通过 | 不可见 |
| 已驳回 | 可查看、编辑、重新提交 | 不可评分 | 可查看历史审核记录 | 不可见 |
| 审核通过 | 可查看结果 | 可查看、可评分 | 可发布、下线、设为精选 | 不可见 |
| 已发布 | 可查看 | 可查看、可参考评分结果 | 可下线、取消精选 | 可见、可浏览、可点赞、可查看评论 |
| 已下线 | 可查看历史状态 | 默认不可见或按权限查看 | 可重新发布或保持下线 | 不可见 |

状态流转测试需重点覆盖以下规则：

- 草稿状态下允许修改和删除，提交后应限制重复提交
- 驳回后的作品允许修改并重新提交，且需保留审核历史
- 未审核通过的作品不得进入评分和公开展示环节
- 已发布作品的删除、编辑、下线、再发布操作必须符合角色和状态限制
- 下线后的作品应立即对公开端不可见

## 11. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| DeepSeek API 不可用 | 内容审核功能阻塞 | fallback 机制（bypass/reject/warning），测试中跳过 |
| Redis 不可用 | 缓存相关功能降级 | 测试中尽量绕过缓存，或 Mock Redis |
| 数据库密码硬编码 | 安全泄露 | 改为 `${DB_PASSWORD}` 环境变量 |
| 无 H2 兼容性验证 | SQL 方言差异 | 确保 SQL 使用 MySQL 标准语法，或直接使用测试 MySQL |
| 时间不足 | 测试覆盖率不够 | 优先覆盖核心业务流（作品→审核→评分→发布） |

## 12. 缺陷管理

### 12.1 缺陷分级

| 级别 | 定义 | 示例 |
|------|------|------|
| `P0` | 核心流程阻断、严重数据错误、严重安全或越权问题 | 无法登录、审核/评分/发布主流程中断、用户可访问他人敏感数据 |
| `P1` | 主要功能异常，但存在替代路径或影响范围可控 | 评论失败、排行榜异常、上传功能偶发失败 |
| `P2` | 一般功能、界面展示或低风险问题 | 文案错误、样式错位、非核心查询异常 |

### 12.2 缺陷记录要求

所有测试中发现的缺陷统一记录在 `docs/缺陷报告.md` 中，按 P0-P2 分级管理。

每个缺陷至少包含以下信息：

- 缺陷标题
- 所属模块
- 测试环境与版本号
- 前置条件
- 复现步骤
- 期望结果
- 实际结果
- 截图、日志或关键报错信息
- 缺陷级别、状态、责任人

### 12.3 缺陷流转要求

- `P0` 缺陷需优先修复并立即回归
- `P1` 缺陷应在本轮测试结束前完成修复评估和回归安排
- `P2` 缺陷可根据发布时间和风险统一排期，但需记录是否接受遗留
- 所有已修复缺陷必须执行回归测试，并记录结果

## 15. 使用说明

### 15.1 标记规则

每完成一项，将 `[ ]` 改为 `[x]` 即可标记为已完成。

### 15.2 文档联动规则

**每完成一个任务（或一组关联任务），必须同步更新以下 4 个文档：**

| 文档 | 更新时机 | 更新内容 |
|------|---------|---------|
| `测试日志.md` | 每次执行测试后 | 追加新的测试轮次，记录日期/范围/结果/失败详情 |
| `测试数据准备.md` | 测试数据发生变化时 | 新增/修改测试账号、SQL、环境配置 |
| `接口测试用例.md` | 新增测试覆盖后 | 在 API 标题旁标记 `[自动]` / `[手动]` / `[未覆盖]` |
| `缺陷报告.md` | 发现新缺陷时 | 按 P0-P2 分级记录，包含复现步骤和环境信息 |

### 15.3 禁止项

- 不允许只改代码不改文档
- 不允许在文档标记 `[自动]` 但实际未编写对应自动化测试
- 不允许跳过 `测试日志.md` 直接合并测试轮次
