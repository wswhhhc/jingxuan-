## 单元测试过程中发现的问题

### BUG-001：MyBatis-Plus 3.5.7 insert 方法存在歧义重载

| 字段 | 值 |
|------|----|
| **级别** | P1 |
| **模块** | 后端 — MyBatis-Plus 兼容性 |
| **涉及文件** | 所有单元测试 |
| **发现时间** | 2026-05-24 |
| **状态** | 已规避 |

**描述：** MyBatis-Plus 3.5.7 在 `BaseMapper` 中新增了 `int insert(Collection<T> entityList)` 批量插入方法。由于 Java 泛型擦除，编译后两个 `insert` 方法的签名均为 `insert(Object)`，导致 Mockito 的 `any()`、`any(Work.class)` 等参数匹配器无法区分应匹配哪个重载，编译时报"方法引用不明确"。

```
BaseMapper 中的方法 insert(T) 和 BaseMapper 中的方法 insert(Collection<T>) 都匹配
```

**影响范围：** 所有需要对 Mapper 的 `insert` 方法进行 stub 或 verify 的单元测试。

**根本原因：** MyBatis-Plus 3.5.7 新增的批量操作方法与原有单实体方法签名在擦除后冲突，Mockito 的类型推断无法处理。

**规避方案：** 使用显式类型转换 `(Work) any()` 帮助编译器选择重载：
```java
// 修复前（编译错误）
when(workMapper.insert(any())).thenReturn(1);
// 修复后
when(workMapper.insert((Work) any())).thenReturn(1);
```

### BUG-002：Wrappers.lambdaUpdate() 在纯 Mockito 测试中不可用

| 字段 | 值 |
|------|----|
| **级别** | P1 |
| **模块** | 后端 — 测试兼容性 |
| **涉及文件** | `WorkServiceImpl.java`（deleteWork 方法） |
| **发现时间** | 2026-05-24 |
| **状态** | 已规避 |

**描述：** `WorkServiceImpl.deleteWork()` 内部调用了 `Wrappers.<WorkAttachment>lambdaUpdate()` 构建更新条件。该方法在运行时会尝试初始化 MyBatis-Plus 的 Lambda 元数据缓存，在纯 Mockito 单元测试（未加载 Spring 上下文）中会抛出 `MybatisPlusException: can not find lambda cache for this entity`。

```
com.baomidou.mybatisplus.core.exceptions.MybatisPlusException:
  can not find lambda cache for this entity [com.jingxuan.entity.WorkAttachment]
```

**影响范围：** 涉及 `Wrappers.lambdaUpdate()` 或 `Wrappers.lambdaQuery()` 的 Service 方法在纯 Mockito 测试中均无法直接执行。

**根本原因：** MyBatis-Plus Lambda 缓存依赖 `Configuration` 初始化，纯 Mockito 环境未加载此配置。

**规避方案：** 在测试中通过 `mockStatic(Wrappers.class)` 模拟 `lambdaUpdate()` 调用，返回预构建的 mock `LambdaUpdateWrapper`：
```java
try (MockedStatic<Wrappers> wrappersMock = mockStatic(Wrappers.class)) {
    LambdaUpdateWrapper mockWrapper = mock(LambdaUpdateWrapper.class);
    when(mockWrapper.set(any(), any())).thenReturn(mockWrapper);
    when(mockWrapper.eq(any(), any())).thenReturn(mockWrapper);
    wrappersMock.when(() -> { Wrappers.lambdaUpdate(); }).thenReturn(mockWrapper);
    workService.deleteWork(1L);
}
```

**长期建议：** 如果测试中频繁遇到此问题，可考虑将 Service 中的 `Wrappers.lambdaXxx()` 调用抽取为 `protected` 方法，以便子类或测试进行覆写。

### BUG-003：SecurityUtils 静态方法依赖导致单元测试需额外 Mock

| 字段 | 值 |
|------|----|
| **级别** | P2 |
| **模块** | 后端 — 可测试性 |
| **涉及文件** | 所有涉及 SecurityUtils 的 Service |
| **发现时间** | 2026-05-24 |
| **状态** | 已有规避方案 |

**描述：** `AuditServiceImpl`、`WorkServiceImpl` 等核心 Service 大量调用 `SecurityUtils.requireCurrentUserId()` 静态方法获取当前用户信息。由于它是静态方法，在单元测试中必须使用 `Mockito.mockStatic()` 才能控制其返回值。

**影响范围：** 所有业务 Service 的单元测试都需要 mock 静态方法，增加了测试编写成本。

**改进建议（可选）：** 可以引入一个非静态的 `SecurityContextProvider` 接口，通过构造器注入，使测试更简洁：

```java
@Component
public class SecurityContextProvider {
    public Long getCurrentUserId() { return SecurityUtils.requireCurrentUserId(); }
}
```

当前已通过 `@BeforeEach` / `@AfterEach` 统一管理 MockedStatic，实现了测试隔离。

### BUG-004：Mockito 嵌套 when() 导致 UnfinishedStubbingException

| 字段 | 值 |
|------|----|
| **级别** | P2 |
| **模块** | 后端 — 测试编写 |
| **发现时间** | 2026-05-24 |
| **状态** | 已修复 |

**描述：** 在 `WorkServiceImplTest.shouldDeleteDraftWork` 中，`wrappersMock.when(() -> { Wrappers.lambdaUpdate(); }).thenReturn(mockLambdaUpdate())` 中的 `mockLambdaUpdate()` 内部调用了 `when(mockWrapper.set(any(), any())).thenReturn(mockWrapper)`，导致 Mockito 在未完成外部 `when()` 设置时开始了新的 stubbing，抛出 `UnfinishedStubbingException`。

```
Unfinished stubbing detected here:
-> at WorkServiceImplTest$DeleteWork.shouldDeleteDraftWork(WorkServiceImplTest.java:325)
Hints:
 1. missing thenReturn()
 2. you are trying to stub a final method, which is not supported
 3. you are stubbing the behaviour of another mock inside before 'thenReturn' instruction is completed
```

**修复：** 将 LambdaUpdateWrapper 的 mock 创建移出 `thenReturn()` 参数，先创建再传入：
```java
// 修复前（嵌套 when）
wrappersMock.when(() -> { Wrappers.lambdaUpdate(); }).thenReturn(mockLambdaUpdate());

// 修复后（先创建 mock，再传入）
LambdaUpdateWrapper mockWrapper = createMockLambdaUpdate();
wrappersMock.when(() -> { Wrappers.lambdaUpdate(); }).thenReturn(mockWrapper);
```
