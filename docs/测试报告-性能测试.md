## 2. 单请求延迟

测试方法：每个端点连续请求 50 次，取算术平均响应时间。

| 接口 | 方法 | 平均响应时间 | 说明 |
|------|------|-------------|------|
| `GET /public/ranking/list` | GET | **4 ms** | 排行榜（Redis 缓存命中） |
| `GET /public/works` | GET | **13 ms** | 作品列表（MySQL 分页查询） |
| `GET /admin/dashboard/stats` | GET | **20 ms** | 仪表盘（含多条统计 SQL） |
| `POST /auth/login` | POST | **85 ms** | 登录（含密码加密比对 + JWT 签发） |

**结论：** 核心接口平均响应时间 < 100ms，满足性能要求。

## 3. 并发压力测试

### 3.1 认证接口（无限流）

测试接口：`GET /admin/dashboard/stats`（需 Bearer Token）

| 并发数 | 总请求 | 吞吐量 | 平均响应时间 | 失败数 | 表现 |
|--------|--------|--------|-------------|--------|------|
| 10 | 50 | 218 req/s | 45.7 ms | 0 | ✅ 稳定 |
| 50 | 100 | 183 req/s | 273.1 ms | 0 | ✅ 稳定 |
| 100 | 100 | 216 req/s | 461.6 ms | 0 | ✅ 稳定 |

**结论：** 即使 100 并发同时请求，服务端仍然 100% 正常响应，无请求失败。平均响应时间在可接受范围内。

### 3.2 公开接口（限流机制验证）

测试接口：`GET /public/ranking/list`

测试配置：30 并发 / 50 请求

| 指标 | 值 |
|------|-----|
| 总请求 | 50 |
| 成功（200） | 19 |
| 限流拒绝（429） | 31 |
| 吞吐量 | 700 req/s |
| 平均响应时间 | 42.8 ms |

**结论：** 公开 API 限流器正常工作。配置为 20 请求/秒/IP，超出部分返回 429。   
失败请求均为预期的限流拒绝，非服务端错误。如需更高并发，可在 `PublicRateLimitFilter` 中调整 `LIMIT` 常量。

## 4. 关键发现

| 发现 | 详情 |
|------|------|
| ✅ 登录接口 | 85ms 是性能瓶颈点（密码加密 + JWT），但单次登录高频场景，可接受 |
| ✅ 排行榜缓存 | 4ms 极低延迟，Redis 缓存命中后几乎无开销 |
| ✅ 并发稳定性 | 100 并发下 0 失败，服务端未出现 OOM 或连接池耗尽 |
| ✅ 限流机制 | 正确的 429 拒绝，保护后端免受突发流量冲击 |
| ℹ️ JVM 配置 | 当前 `-Xmx2g`，在 100 并发下内存充足，未触发 GC 瓶颈 |

## 6. 改进建议

| 建议 | 预期收益 | 难度 |
|------|---------|------|
| 登录接口添加缓存（短时间相同账号不重复加密比对） | 减少 50ms 延迟 | 低 |
| 仪表盘统计数据加 Redis 缓存（目前为实时 SQL） | 响应从 20ms 降至 <5ms | 低 |
| Tomcat 线程池从默认 200 调整为 400（当前已配） | 更高并发承载 | 无需改动 |
| 分库或读写分离（MySQL 同机压力） | 突破数据库瓶颈 | 高 |

---

## 7. 测试局限性

- 测试工具与被测服务运行在同一台机器，网络延迟几乎为 0，实际部署时存在网络开销
- 未测试长时间稳定性（如持续 30 分钟高压）
- 未测试慢查询场景（百万级数据量下的排行榜）
- 未测试文件上传并发（涉及磁盘 IO）
