## 1. 数据初始化顺序

数据库初始化必须按以下顺序执行 SQL：

```
1. backend/sql/base/init_schema.sql          — 基础表（8 张：用户、角色、菜单等）
2. backend/sql/business/work_schema.sql      — 业务表（12 张：作品、评分、审核等）
3. backend/sql/business/test_data.sql        — 测试种子数据
4. backend/sql/business/2026-05-19-fix-work-attachment-nullable.sql  — 迁移1
5. backend/sql/business/2026-05-24-like-fav-view-tag.sql             — 迁移2
```

### 注意事项

- 基础表 `init_schema.sql` 包含 `sys_user`、`sys_role`、`sys_menu`、`sys_role_menu`、`sys_dict`、`sys_log`、`sys_notice`、`sys_notification` 共 8 张表
- 业务表 `work_schema.sql` 包含 work、work_member、work_attachment、work_audit、work_publish、work_score、work_comment、score_batch、sensitive_rule、rank_reward、reward_issue、port_manage、reward_config 共 13 张表（含 reward_config 别名 rank_reward）
- 迁移 SQL 必须在建表之后执行，它们修改表结构或追加数据
- 迁移 SQL 中 `work_like`、`tag`、`work_tag` 三张表由迁移 2 创建，非基础建表脚本

### 全量初始化脚本（供参考）

```sql
-- 按顺序执行
SOURCE backend/sql/base/init_schema.sql;
SOURCE backend/sql/business/work_schema.sql;
SOURCE backend/sql/business/test_data.sql;
SOURCE backend/sql/business/2026-05-19-fix-work-attachment-nullable.sql;
SOURCE backend/sql/business/2026-05-24-like-fav-view-tag.sql;
```

## 3. 数据依赖关系

### 3.1 表关系图

```
sys_user (学生) ──→ work (submitter_id) ──→ work_publish
                     work (batch_id) ──→ score_batch
                     work (id) ──→ work_audit
                     work (id) ──→ work_score ──→ sys_user (教师)
                     work (id) ──→ work_member ──→ sys_user (学生成员)
                     work (id) ──→ work_comment ──→ sys_user (评论人)
                     work (id) ──→ work_like ──→ sys_user (点赞人)
                     work (id) ──→ work_tag ──→ tag
                     work (id) ──→ port_manage
score_batch ──→ reward_config ──→ reward_issue ──→ work
```

### 3.2 关键依赖约束

| 约束 | 说明 | 违反后果 |
|------|------|---------|
| work.submitter_id → sys_user.id | 提交者必须是已存在的用户 | 外键错误或查询 null |
| work.batch_id → score_batch.id | 作品关联的批次必须存在 | 评分时找不到批次 |
| work_score.teacher_id → sys_user.id | 评分教师必须存在 | 评分查询时 teacherName 为 null |
| work_publish.work_id → work.id | 发布记录关联的作品必须存在 | 唯一约束冲突或数据不一致 |
| uk_work_teacher (work_id, teacher_id) | 同一教师对同一作品只能评分一次 | 违反唯一约束 |
| uk_work_user (work_id, user_id) | 同一用户对同一作品只能点赞一次 | 违反唯一约束 |
| uk_work_tag (work_id, tag_id) | 同一作品同一标签只能关联一次 | 违反唯一约束 |

### 3.3 作品状态与可用操作

| 状态(status) | 含义 | 学生可操作 | 教师可操作 | 管理员可操作 |
|-------------|------|-----------|-----------|------------|
| 0 | 草稿 | 查看/编辑/删除/提交 | — | 查看 |
| 1 | 已提交(待审核) | 查看(不可编辑) | — | 审核(通过/驳回) |
| 2 | 已驳回 | 查看/编辑/重新提交 | — | 查看 |
| 3 | 已通过 | 查看 | 评分/查看详情 | 发布/设为精选/下线 |

### 3.4 发布状态(publish_status)与作品状态的关系

```
审核通过(work.status=3)
    → 发布(work_publish.publish_status=1)    → 前台可见
    → 下线(work_publish.publish_status=2)    → 前台不可见
    → 未发布(work_publish.publish_status=0)  → 前台不可见
```

只有 `work.status=3`（已通过）且 `work_publish.publish_status=1`（已发布）的作品对公开端可见。

### 4.2 常用测试数据构造（Java 参考）

```java
// 作品构造
private Work createWork(Long id, int status, Long submitterId, Long batchId) {
    Work work = new Work();
    work.setId(id);
    work.setTitle("测试作品");
    work.setSummary("测试简介");
    work.setTechStack("Java/Spring Boot");
    work.setStatus(status);
    work.setSubmitterId(submitterId);
    work.setBatchId(batchId);
    work.setSubmitTime(LocalDateTime.now());
    return work;
}

// 评分请求构造
private ScoreSubmitRequest createValidRequest() {
    ScoreSubmitRequest req = new ScoreSubmitRequest();
    req.setWorkId(1L);
    req.setInnovation(new BigDecimal("20"));
    req.setDifficulty(new BigDecimal("20"));
    req.setCompletion(new BigDecimal("25"));
    req.setPracticality(new BigDecimal("15"));
    req.setComment("做得好");
    return req;
}

// 批次构造
private ScoreBatch createBatch(Long id, int status) {
    ScoreBatch batch = new ScoreBatch();
    batch.setId(id);
    batch.setBatchName("测试批次");
    batch.setStatus(status);
    batch.setStartTime(LocalDateTime.now().minusDays(1));
    batch.setEndTime(LocalDateTime.now().plusDays(1));
    return batch;
}
```

### 4.3 Mock 策略

| 依赖 | Mock 策略 | 说明 |
|------|----------|------|
| Mapper | `@Mock` + `when().thenReturn()` | MyBatis-Plus BaseMapper 方法需注意类型推断（3.5.7+ 有 `insert(Collection)` 重载，需显式类型转换） |
| HttpClient | `@Mock` + `ReflectionTestUtils.setField()` | AiUserImportServiceImpl 中 HttpClient 为 inline 初始化，需用反射替换 |
| Redis | 测试环境跳过 | `application-test.yml` 中保留连接配置但测试代码 Mock Redis 操作 |
| DeepSeek API | `fallback=bypass` 或 Mock HTTP 调用 | 单元测试中完全 Mock 外部 API |
| 文件上传 | `TestRestTemplate` + `ByteArrayResource` + `MultiValueMap` | 模拟 multipart/form-data 请求，需手动设置 `Content-Type: multipart/form-data` 和 Bearer token |
| 并发测试 | `ExecutorService` + `Future` | 用 5 线程池提交 10 次并发请求，验证无死锁/数据不一致 |

## 5. 集成测试数据准备

### 5.1 环境配置

集成测试使用 `application-test.yml`，关键配置：

```yaml
# H2 内存数据库（MySQL 兼容模式）
spring.datasource.url: jdbc:h2:mem:jingxuan_test;MODE=MySQL
# DeepSeek 审核绕过
deepseek.fallback: bypass
# 日志降噪
logging.level.com.jingxuan: warn
```

### 5.2 集成测试专用数据

集成测试使用 H2 兼容的 `test-schema.sql` + `test-data.sql`（位于 `backend/src/test/resources/sql/`），与生产 SQL 的关键差异：

| 差异项 | 生产 SQL | 测试 SQL |
|--------|---------|---------|
| 数据库 | MySQL 8.0 | H2 MySQL 兼容模式 |
| 语法 | `USE jingxuan;` / `ON DUPLICATE KEY` | 已移除 MySQL 专有语法 |
| 迁移 | 独立增量 SQL | 合并在 `test-schema.sql` 建表 |
| view_count/like_count | 迁移 SQL 追加 | 建表时已含 |
| tag/work_like/work_tag | 迁移 SQL 创建 | 建表时已含 |
| 清理策略 | 手动 | `test-data.sql` 开头 `DELETE FROM` + `cleanup.sql` |
| 测试学生 | 不含 | 含 `teststu`(ID=110, 无关联作品, 适合 CRUD) |

### 5.3 测试数据注入方式

推荐使用 Spring 的 `@Sql` 注解按顺序执行 SQL：

```java
@SpringBootTest
@Sql(executionPhase = Sql.ExecutionPhase.BEFORE_TEST_CLASS, scripts = {
    "classpath:sql/base/init_schema.sql",
    "classpath:sql/business/work_schema.sql",
    "classpath:sql/business/test_data.sql",
    "classpath:sql/business/2026-05-19-fix-work-attachment-nullable.sql",
    "classpath:sql/business/2026-05-24-like-fav-view-tag.sql"
})
@Sql(executionPhase = Sql.ExecutionPhase.AFTER_TEST_CLASS, scripts = {
    "classpath:sql/cleanup.sql"
})
class MyIntegrationTest {
    // ...
}
```

### 5.3 @Sql 脚本路径说明

使用时需将 SQL 文件复制到测试资源目录：

```
backend/src/test/resources/sql/
  base/init_schema.sql
  business/work_schema.sql
  business/test_data.sql
  business/2026-05-19-fix-work-attachment-nullable.sql
  business/2026-05-24-like-fav-view-tag.sql
  cleanup.sql
```

或修改 `@Sql` 路径为相对于 classpath 的绝对路径。

### 5.4 集成测试事务策略

```java

// 避免测试数据相互污染
```

## 6. 环境配置

### 6.1 `.env` 文件

项目根目录下已有 `.env` 文件，包含 DeepSeek API Key：

```
DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

**严禁将包含真实 API Key 的 `.env` 文件提交到 Git！**

### 6.2 缺失的 `.env.test`

测试环境暂缺 `.env.test` 文件，建议创建：

```
# 测试环境 DeepSeek API Key（可用测试或空值配合 fallback=bypass）
DEEPSEEK_API_KEY=sk-test-key
```

### 6.3 application-dev.yml 密码配置

当前 `application-dev.yml` 中数据库密码为明文硬编码。建议改为环境变量引用：

```yaml
spring.datasource.password: ${DB_PASSWORD:root}
```

同时创建 `.env` 或系统环境变量：

```
DB_PASSWORD=root
```
