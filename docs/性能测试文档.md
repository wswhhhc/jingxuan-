## 1. 性能测试目标

### 1.1 测试范围

| 层次 | 测试内容 | 工具/方法 |
|------|---------|----------|
| API 响应时间 | 核心接口的响应耗时 | JMeter / Postman |
| 并发能力 | 多用户同时操作下的吞吐量 | JMeter 线程组 |
| 数据库查询 | 复杂查询/大批量数据下的 SQL 效率 | MySQL EXPLAIN / H2 分析 |
| 前端加载 | 页面首次渲染和资源加载时间 | Lighthouse / Chrome DevTools |
| 文件上传 | 大文件上传耗时和内存占用 | 手动测试 + 监控 |

### 1.2 性能基线（目标值）

| 接口 | 平均响应时间 | 95 分位 | 吞吐量 |
|------|------------|---------|--------|
| 公开作品列表（GET /public/works） | < 500ms | < 1000ms | ≥ 100 req/s |
| 作品详情（GET /public/works/{id}） | < 300ms | < 500ms | ≥ 200 req/s |
| 用户登录（POST /auth/login） | < 500ms | < 800ms | ≥ 50 req/s |
| 提交评分（POST /teacher/score） | < 500ms | < 800ms | ≥ 50 req/s |
| 排行榜（GET /public/ranking/list） | < 800ms | < 1500ms | ≥ 50 req/s |
| 文件上传（POST /api/file/upload） | < 3000ms | < 5000ms | ≥ 10 req/s |

## 5. 性能测试执行

### 5.1 前置步骤

1. 准备测试数据（使用 `backend/sql/performance/test_data.sql` 或 JMeter 前置脚本）
2. 确保 DeepSeek 审核设置为 `fallback=bypass` 避免外部依赖影响
3. 预热：先执行 100 次请求使 JIT 和缓存达到稳定状态
4. 记录基准环境信息（CPU / 内存 / JVM 参数）

### 5.2 执行顺序

1. **基准测试**：单用户单接口，获取基线响应时间
2. **负载测试**：逐步增加并发数（10 → 50 → 100 → 200）
3. **压力测试**：在 200 并发下运行 5 分钟
4. **稳定性测试**：100 并发混合场景运行 30 分钟
5. **尖峰测试**：突发 200 并发持续 30s，观察系统恢复

## 6. 性能优化建议

### 6.1 数据库

| 问题 | 建议 | 预期效果 |
|------|------|---------|
| 排行榜查询涉及多表 JOIN | 使用 Redis 缓存排行榜结果 | 响应时间从 500ms → 10ms |
| 作品列表含子查询 | 增加 `view_count`/`like_count` 冗余字段 | 避免 COUNT 子查询 |
| 标签筛选多表关联 | 使用 `work_tag` 索引覆盖 | 索引生效后减少回表 |
| 审核列表 WHERE status=1 | 复合索引 `(status, submit_time)` | 筛选效率提升 |

### 6.2 应用层

| 问题 | 建议 | 预期效果 |
|------|------|---------|
| 每次请求解析 JWT | 增加 Token 本地缓存验证 | 减少重复解析开销 |
| 频繁的 `selectById` | 批量查询替换循环查询 | N+1 → 1 次查询 |
| 敏感词审核同步调用 | 改为异步队列处理 | 提交审核响应时间降低 |
| 图片/文件未缓存 | 配置 CDN 或 Nginx 静态缓存 | 文件加载速度提升 |

### 6.3 Redis 缓存策略

| 缓存项 | 有效期 | 更新时机 |
|--------|--------|---------|
| 排行榜 | 5 分钟 | 新评分提交后刷新 |
| 作品详情 | 2 分钟 | 更新作品/评论后失效 |
| 标签列表 | 10 分钟 | 标签增删后失效 |
| 班级列表 | 30 分钟 | 字典更新后失效 |
