# 学院作品展示平台 需求文档 V1.9

## 一、项目概述

### 1.1 项目背景
某 IT 学院每年会产生大量学生课程作品、实训项目与毕业设计成果。传统的作品提交、展示、评分和汇总主要依赖人工收集与线下管理，存在资料分散、统计困难、展示效果弱、评审效率低等问题。为提升学院作品管理与展示水平，拟开发一套线上作品展示平台，实现作品提交、审核、展示、评分、排行与精选发布的一体化管理。

> **与原始需求的差异说明**：相较学院原始课题描述，本文档新增了"管理员审核"环节。原始需求中学生提交后直接进入展示，增加审核环节的目的是：① 确保展示作品信息完整、附件有效；② 避免不合规内容直接暴露在前台；③ 在实际教学场景中更贴近真实上线流程。此改动已在指导老师处确认。

### 1.2 项目目标
- 提供统一的学生作品提交入口，支持压缩包、截图、演示视频等资料上传
- 支持作品图文展示、视频播放与少量精选作品在线体验
- 建立教师评分与评语机制，支持排行榜统计与展示
- 提供管理员审核、公告发布、LLM 内容安全审核规则配置、日志追踪等后台能力
- 满足课程实训项目的完整性、可演示性与可维护性要求

### 1.3 建设原则
- 以 PC 端为主，优先保证教师评审与后台管理体验
- 以“作品”为核心对象，兼顾个人作品与团队作品
- 采用前后端分离架构，便于分工协作与后续扩展
- 坚持“先完成核心闭环，再实现加分功能”的开发策略

### 1.4 用户角色

| 角色 | 职责 |
|------|------|
| 学生 | 提交作品、补充资料、查看审核结果、查看评分与排行 |
| 教师 | 浏览作品、查看详情、评分、填写评语 |
| 管理员 | 用户管理、作品审核、发布/下线、精选与预览地址维护、内容审核规则管理、公告发布、日志查看 |

## 二、业务范围与流程

### 2.1 业务范围
本系统主要服务于学院内部课程作品、实训作品和毕业作品的集中管理与展示，不面向社会化投稿与公开运营。系统重点解决以下问题：
- 学生作品资料统一提交与管理
- 教师线上评分与评语记录
- 优秀作品的集中展示与排行公示
- 少量精选作品的在线体验部署

### 2.2 核心业务流程
1. 管理员导入或创建用户账号，配置基础数据
2. 学生登录系统后填写作品信息并提交作品资料
3. 管理员审核作品内容与附件完整性
4. 审核通过的作品进入教师评审区，教师可进行评分与评语填写
5. 系统根据评分规则生成排行榜
6. 管理员从已发布作品中挑选 1 至 2 个精选作品，配置已部署项目的服务器访问地址

### 2.3 作品状态与发布状态

作品状态分为“审核状态”和“发布状态”两类，分别对应不同业务职责，避免同一个字段混合表达审核与展示含义。

#### 2.3.1 审核状态流转

| 状态 | 说明 | 可执行角色 |
|------|------|------|
| 草稿 | 学生已创建但未正式提交 | 学生 |
| 已提交 | 学生已提交，等待管理员审核（学生只读，不可编辑） | 管理员 |
| 已驳回 | 管理员审核未通过，需学生修改后重新提交 | 学生、管理员 |
| 已通过 | 管理员审核通过，教师可评分；前台暂不可见 | 管理员、教师 |

说明：
- 学生仅可编辑“草稿”“已驳回”状态下的作品
- 学生仅可删除“草稿”状态或“已驳回且未重新提交”状态的作品
- 教师仅对审核状态为“已通过”的作品进行评分；作品是否已发布不影响教师评分权限
- “已通过”表示作品已通过审核，但不等于前台公开展示

#### 2.3.2 发布状态流转

| 状态 | 说明 | 可执行角色 |
|------|------|------|
| 未发布 | 作品审核通过后尚未在前台展示 | 管理员 |
| 已发布 | 管理员手动发布后，作品在前台公开可见 | 管理员、教师、学生 |
| 已下线 | 展示结束或管理员手动下线，前台不可见 | 管理员 |

说明：
- “已通过”与“已发布”的区别：审核通过后作品进入“已通过”状态，此时教师端可见、可评分，但学生前台不可见；管理员执行“发布”操作后，发布状态变为“已发布”，作品才在前台公开展示。两个状态分离的目的是让管理员在审核通过后还有一次展示控制机会（如统一发布时间、暂缓展示等）
- 精选展示不是独立状态，而是在已发布作品上设置 `featured=1` 标记；只有精选作品可配置在线体验地址

## 三、功能需求

### 3.1 用户管理模块

| 编号 | 功能 | 描述 |
|------|------|------|
| U-01 | 账号导入/创建 | 管理员可按学号、工号批量导入或手动创建学生、教师账号 |
| U-02 | 用户登录 | 用户通过账号密码登录系统 |
| U-03 | 首次登录改密 | 首次登录后需修改初始密码 |
| U-04 | 个人信息维护 | 用户可修改头像、联系方式、密码等信息 |
| U-05 | 角色权限管理 | 管理员可分配角色并控制菜单与接口权限 |
| U-06 | 账号状态管理 | 管理员可启用、禁用账号 |

说明：
- 出于校园场景安全性考虑，优先采用“管理员预置账号”的方式，而非开放式自助注册
- “记住密码”建议调整为“记住登录状态”

### 3.2 作品管理模块

| 编号 | 功能 | 描述 |
|------|------|------|
| W-01 | 创建作品 | 学生创建作品，填写名称、简介、技术栈、指导教师、封面图等信息 |
| W-02 | 附件上传 | 上传作品压缩包、截图、演示视频或视频链接、运行说明 |
| W-03 | 团队成员维护 | 支持维护团队成员信息，标记队长 |
| W-04 | 作品提交 | 学生确认资料完整后提交审核 |
| W-05 | 作品编辑 | 在允许状态下修改作品信息或重新上传附件 |
| W-06 | 作品删除 | 删除未正式进入审核流程的作品 |
| W-07 | 状态查询 | 查看审核状态、驳回原因、发布时间等信息 |

### 3.3 团队作品规则
- 系统同时支持个人作品与团队作品
- 每个作品只能有 1 名负责人（队长）发起提交
- 队长可维护团队成员名单，成员信息至少包含姓名、学号、班级
- 同一学生在同一评审批次中默认只能归属 1 个作品（仅对已注册账号生效；未注册成员因无法验证身份，不纳入此约束，属于有意的设计折中）
- 排行榜按”作品”维度统计，不按个人维度重复计分

### 3.4 审核与展示模块

| 编号 | 功能 | 描述 |
|------|------|------|
| D-01 | 作品审核 | 管理员审核作品信息与资料完整性，可通过或驳回 |
| D-02 | 作品列表 | 按班级、技术栈、提交时间、状态等条件筛选浏览 |
| D-03 | 作品详情 | 展示作品名称、团队信息、简介、技术栈、截图、视频、运行说明 |
| D-04 | 视频播放 | 支持在线播放演示视频 |
| D-05 | 作品搜索 | 按作品名称、学生姓名、技术栈等关键词搜索 |
| D-06 | 前台发布 | 审核通过后由管理员控制是否公开展示 |
| D-07 | 作品下线 | 管理员将已发布作品下线，下线后前台不可见 |

说明：
- 教师评分页面采用匿名评审模式，仅展示作品名称、简介、技术栈、截图、视频等信息
- 前台公开展示页和管理员详情页可展示团队成员信息

### 3.5 评分与评论模块

| 编号 | 功能 | 描述 |
|------|------|------|
| S-01 | 教师评分 | 教师对作品按多个维度打分 |
| S-02 | 教师评语 | 教师填写评语，支持在截止前修改 |
| S-03 | 评分记录查询 | 管理员可查看评分明细与汇总结果 |
| S-04 | 评论互动 | 学生和教师可在作品下方进行交流评论 |
| S-05 | LLM 内容安全审核 | 对评语、评论、作品简介等文本进行内容安全检测，调用 DeepSeek 官方 API（当前配置模型 `deepseek-chat`）做 LLM 内容审核，识别色情、暴力、辱骂、政治敏感、广告引流等违规内容 |
| S-06 | 检测规则管理 | 管理员可维护审核 prompt 模板、违规类别开关、检测失败时的处理策略（拒绝/警告/人工复核） |
| S-07 | 评分批次管理 | 管理员创建评分批次，设置评分起止时间、适用班级范围；批次过期后教师不可再提交或修改评分 |

**内容安全审核实现方案（LLM 方式）：**

调用链路：
```
用户提交文本 → 后端收到请求
  → 调用 DeepSeek 官方 API（base_url: https://api.deepseek.com，path: /chat/completions）
  → 模型: deepseek-chat
  → system prompt 定义审核规则（违规类别、输出格式）
  → 设置 response_format: {"type":"json_object"}
  → 返回 JSON: {"passed": true/false, "category": "违规类别", "reason": "具体原因"}
  → passed=true → 正常入库
  → passed=false → 拒绝提交，将原因返回前端提示用户修改
```

相比传统关键词匹配方案的优势：
- 能识别谐音、变体、隐晦表达，不会被简单绕过
- 不需要手动维护大量违禁词库，减少教师和管理员维护成本
- 新增违规类别只需修改 prompt，无需改代码

注意事项：
- 每次检测有 1-3 秒网络延迟，前端需做 loading 提示
- API 按 token 计费，单次检测约消耗 200-500 token
- 需对 JSON 解析失败、接口超时、API 不可用等情况做兜底处理（重试或进入人工复核）

### 3.6 评分规则
- 评分维度共 4 项，总分 100 分
- 建议评分占比如下：
  - 创新性：25 分
  - 技术难度：25 分
  - 完成度：30 分
  - 实用性：20 分
- 每个教师对同一作品只能提交 1 次正式评分，但可在评分截止前修改
- 若一个作品由多位教师评分，则最终得分取所有教师总分的平均值，保留两位小数
- 若排行榜分数相同，则依次按“完成度得分”“创新性得分”“作品提交时间（submit_time）”进行排序，提交时间早者排前
- 评分批次由管理员在后台创建，一个批次可关联多个班级；同一作品若跨批次展示，以最新批次评分为准
- 教师评分时默认隐藏学生姓名与学号，仅展示作品名称与简介（避免人情分）；管理员可查看评分人与被评作品的完整对应关系
- 学生查看分数与排名：在管理员发布排行榜（R-05）之前，学生不可见任何评分信息；排行榜公示后，学生仅可看到自己的分数与排名

### 3.7 在线体验与部署模块

| 编号 | 功能 | 描述 |
|------|------|------|
| P-01 | 精选作品标记与筛选 | 管理员从已发布作品中选取 1 至 2 个标记为精选，作为在线体验对象 |
| P-02 | 预览地址维护 | 管理员为精选作品维护已部署项目的 `previewUrl` 服务器访问地址 |
| P-03 | 在线体验入口 | 前台作品详情页在存在 `previewUrl` 时展示“在线体验”按钮 |
| P-04 | 反向代理配置 | 需要统一域名时，通过 Nginx 将 `/preview/{作品ID}` 代理到实际服务 |
| P-05 | 上线/下线管理 | 管理员通过作品发布、下线和精选开关控制是否展示在线体验入口 |

说明：
- 精选作品数量建议控制在 1 至 2 个，避免部署维护成本过高
- 其余作品统一采用“截图 + 演示视频”的方式展示
- **范围收窄说明**：原始课题要求所有作品均可在线操作体验，本文档将其限定为仅精选作品支持在线体验。原因：① 实训服务器资源（端口、内存、磁盘）有限，无法同时部署上百个学生项目；② 每个项目的运行环境各异（JDK 版本、Node 版本、数据库依赖等），统一部署运维成本远超课程范围；③ 精选 1-2 个最优作品已可充分展示平台的核心能力。此简化已与指导老师确认
- 当前落地方式进一步收敛为“管理员配置已部署服务地址”，平台不负责自动解压、构建、启动或停止学生项目进程，也不维护运行端口池

部署策略：
- 精选作品：由管理员或运维人员单独部署到服务器，通过 Nginx 反向代理实现 `http://平台地址/preview/{作品ID}` 在线访问，或直接维护完整外部访问 URL
- 普通作品：仅展示简介、截图、演示视频、运行说明

Nginx 配置示例：

```nginx
location /preview/5 {
    proxy_pass http://127.0.0.1:9003;
    proxy_set_header Host $host;
}
```

### 3.8 排行榜模块

| 编号 | 功能 | 描述 |
|------|------|------|
| R-01 | 综合排行榜 | 按作品最终得分降序排列，默认展示前 10 名 |
| R-02 | 分类排行 | 按技术栈、班级等维度生成排行榜 |
| R-03 | 奖项配置 | 管理员可配置一等奖、二等奖、三等奖及优秀奖的名称与名额 |
| R-04 | 奖品管理 | 管理员可为每个奖项配置奖品名称、图片、数量；系统记录奖品发放状态（未发放/已发放） |
| R-05 | 排行公示 | 前台公开展示排行榜结果，前 10 名标注获奖等级与对应奖品 |
| R-06 | 数据刷新 | 评分更新后自动或手动刷新排行榜 |
| R-07 | 批次筛选 | 前台可按评分批次/学期筛选查看历史排行榜，默认展示当前批次 |

说明：
- 排行榜以"评分批次"为维度独立计算，不同批次（学期）的排行榜互不干扰
- 若奖品实际发放走线下流程，R-04 可简化为"奖项说明"（仅展示奖品名称文字），具体与指导老师确认

## 四、非功能需求

### 4.1 性能要求
- 普通页面在校园网络环境下平均加载时间不超过 3 秒
- 单次上传请求不超过 1600MB，单文件不超过 1536MB；课程验收建议仍控制普通作品压缩包在 200MB 以内，便于备份和演示
- 支持至少 100 人并发访问作品列表、作品详情等常用页面
- 排行榜查询应在 2 秒内返回结果

### 4.2 安全要求
- 用户密码采用 BCrypt 加密存储
- 使用 Spring Security + JWT 完成身份认证与权限控制
- 文件上传需校验扩展名（白名单：zip/rar/7z/jpg/png/gif/mp4/pdf，禁止可执行文件如 exe/sh/bat）、大小、MIME 类型与存储路径，上传后重命名文件防止路径穿越；后端 multipart 当前配置为单文件 1536MB、请求 1600MB
- 对常见 SQL 注入、XSS 等风险进行防护
- 对作品简介、评论、评语等文本进行 LLM 内容安全审核
- 精选作品在线体验地址应仅允许管理员配置，不允许学生自行指定服务地址
- DeepSeek API Key 优先通过环境变量配置，`application.yml` 中仅保留占位配置，不暴露给前端；后台管理界面提供连通性测试按钮，不展示完整 Key

### 4.3 可用性要求
- 系统界面简洁清晰，符合校园管理系统风格
- 以 PC 端为主，兼容 1366 x 768 及以上主流分辨率
- 支持 Chrome、Edge、Firefox 等主流浏览器
- 核心功能操作路径清晰，常见操作不超过 3 次点击

### 4.4 可维护性要求
- 后端采用分层架构，便于模块化开发与维护
- 前后端分离，接口遵循 RESTful 风格
- 数据库设计规范，提供完整建表 SQL 脚本
- 关键业务模块应具备基础注释、接口文档与测试数据

## 五、技术选型建议

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端框架 | Spring Boot 3.x | 课程常用框架，生态完善 |

| ORM | MyBatis-Plus | 提高常规 CRUD 开发效率 |
| 数据库 | MySQL 8.0 | 存储业务数据 |

## 六、数据库核心表设计（建议）

| 表名 | 说明 | 核心字段 |
|------|------|------|
| sys_user | 用户表 | id, username, password, real_name, role_id, class_id, status, first_login |
| sys_role | 角色表 | id, role_name, role_code |
| sys_menu | 菜单权限表 | id, menu_name, parent_id, path, permission |
| work | 作品主表 | id, title, summary, tech_stack, advisor, cover_url, status, submitter_id, submit_time, batch_id |
| work_member | 作品成员表 | id, work_id, student_id, student_name, student_no, class_name, is_leader |
| work_attachment | 作品附件表 | id, work_id, file_name, file_type, file_size, file_url |
| work_audit | 作品审核表 | id, work_id, auditor_id, result, reason, audit_time |
| work_score | 作品评分表 | id, work_id, teacher_id, innovation, difficulty, completion, practicality, total, batch_id |
| work_comment | 作品评论表 | id, work_id, user_id, content, create_time |
| work_publish | 作品发布表 | id, work_id, publish_status, featured, publish_time, offline_time |
| score_batch | 评分批次表 | id, batch_name, start_time, end_time, class_scopes, status |
| sensitive_rule | 内容审核规则表 | id, rule_name, system_prompt, enabled_categories, on_reject_action, create_time |
| rank_reward | 奖项配置表 | id, batch_id, reward_level, reward_name, prize_name, quota |
| reward_issue | 奖品发放记录表 | id, reward_id, work_id, issue_status, issue_time |
| sys_notice | 公告表 | id, title, content, publisher_id, publish_time |
| sys_notification | 消息通知表 | id, user_id, title, content, is_read, create_time |
| sys_log | 操作日志表 | id, user_id, action, target, ip, create_time |
| sys_dict | 数据字典表 | id, dict_type, dict_label, dict_value, sort |

> **work_member 成员说明**：`student_id` 关联 `sys_user.id`（成员在系统中有账号时填入，无账号可为空）；`student_name`、`student_no`、`class_name` 必填，确保未注册账号的团队成员也能被记录。
>
> **班级数据一致性说明**：作品成员中存在两套班级来源——`sys_user.class_id`（字典引用）和 `work_member.class_name`（自由文本）。处理规则：已注册学生（`student_id` 非空）以 `sys_user.class_id` 为准，筛选和统计均走 `sys_user` JOIN，`work_member.class_name` 仅作冗余记录；未注册成员（`student_id` 为空）使用 `work_member.class_name`，仅用于详情页展示，不参与筛选逻辑。

> **status 与 publish_status 职责分离**：`work.status` 仅管理审核流转（草稿→已提交→已驳回→已通过），不可设为“已发布”“已下线”等值；`work_publish.publish_status` 仅管理发布流转（已发布/已下线），不可记录审核状态。`work_publish.featured` 标记精选（0=普通 1=精选）。两个字段通过 work_id 关联，职责互不交叉。

## 九、验收标准建议

### 9.1 功能验收
- 学生可成功登录并提交作品资料
- 管理员可完成审核并查看审核记录
- 教师可对通过审核的作品进行评分与填写评语
- 系统可根据评分结果生成排行榜
- 前台可正常浏览作品列表、详情、视频与排行榜

### 9.2 数据验收
- 用户、作品、评分、公告等核心数据可正确入库；如实现评论功能，评论数据可正确入库
- 作品状态流转符合业务规则
- 排行榜计算结果与评分数据一致

### 9.3 演示验收
- 提供至少 1 个完整的学生提交流程演示
- 提供至少 1 个管理员审核流程演示
- 提供至少 1 个教师评分与排行榜更新演示
- 如实现精选在线体验，应提供可访问的预览地址

## 十一、项目风险与应对

| 风险 | 说明 | 应对策略 |
|------|------|------|
| 功能范围过大 | 实训周期有限，容易后期来不及 | 明确 P0、P1、P2 范围，先完成核心功能 |
| 在线体验实现复杂 | 涉及 Nginx、运行环境和学生项目依赖 | 精选作品限量部署；平台只维护预览地址，必要时降级为静态展示或演示视频 |
| 团队协作冲突 | 前后端接口或字段不统一 | 提前制定接口文档与命名规范 |
| 数据准备不足 | 演示时缺少测试数据 | 提前准备学生、教师、作品、评分样例数据 |
| LLM 接口不稳定 | 网络波动或第三方接口异常会影响内容审核 | 设置超时、重试与人工复核兜底策略 |
